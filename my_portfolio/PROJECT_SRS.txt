1. PROJECT OVERVIEW
======================================================================

1.1 Purpose
-----------
This project is a full-stack dynamic portfolio platform built around a
cinematic, continuous-scroll presentation. The public portfolio is intended to
feel less like a traditional multi-section website and more like an immersive
digital trailer: one seamless visual sequence with shared atmospheric layers,
scroll-linked motion, parallax, and progressive section transitions.

The system combines:
- A public animated portfolio experience.
- A protected admin dashboard.
- PostgreSQL-backed portfolio content.
- Auth.js/NextAuth credentials authentication.
- Human-readable shareable portfolio links.
- Prisma ORM for database modeling and queries.

1.2 Vision
----------
The long-term vision is to evolve this portfolio into a reusable personal
branding platform where the owner can edit content, publish customized public
portfolio links, track basic visits, and maintain a premium cinematic interface
without manually changing frontend code.

The visual direction is inspired by Rockstar / GTA VI style web presentations:
- immersive visual continuity
- dramatic motion language
- cinematic layers
- atmospheric gradients and light leaks
- one-page storytelling
- premium visual pacing

1.3 Target Audience
-------------------
Primary users:
- Portfolio owner / administrator.
- Recruiters, collaborators, clients, and visitors viewing the public portfolio.

Secondary users:
- Developers maintaining or extending the codebase.
- Future contributors implementing analytics, richer CMS editing, multi-user
  features, or media storage.

1.4 Major Capabilities
----------------------
Implemented capabilities include:
- Public portfolio rendering from database-backed content with fallback data.
- Responsive animated portfolio sections:
  home, services, about, projects, education, gallery, and contact.
- GSAP ScrollTrigger animation system.
- Custom inertia-style smooth scrolling.
- Protected admin dashboard.
- Credentials signup/login/logout.
- Prisma schema for Auth.js and CMS data.
- Human-readable share-link routes and visit tracking.
- Vercel deployment configuration.

2. SYSTEM GOALS
======================================================================

2.1 Product Goals
-----------------
- Present the portfolio as a cinematic scroll experience rather than a static
  website.
- Make all public content editable through admin-controlled data.
- Keep the public portfolio accessible without login.
- Restrict dashboard access to authenticated users.
- Support branded shareable URLs that are easy to read and remember.
- Preserve a scalable data model for future multi-user expansion.

2.2 Experience Goals
--------------------
- Sections should visually connect through overlapping transitions.
- Backgrounds should feel global rather than section-specific.
- Animation should use a consistent motion language.
- Desktop should preserve a cinematic full-width experience.
- Mobile should retain accessible navigation through a hamburger menu and
  slide-in drawer.

2.3 Scalability Goals
---------------------
- Keep backend logic modular through services and server actions.
- Keep database relations normalized around User and PortfolioProfile.
- Keep animation logic isolated in reusable hooks.
- Keep Prisma queries selective and ordered.
- Prepare for future richer dashboard modules without changing the public UI.

- useSmoothScroll controls wheel/touch interpolation.

7. DATABASE DESIGN
======================================================================

7.1 Overview
------------
The database is modeled around User ownership. A User owns one
PortfolioProfile. The profile owns all editable portfolio content. ShareLink
belongs to User because it represents publishing/visibility controlled by the
account owner.

7.2 Models
----------

User
- Purpose: Authenticated account owner.
- Fields: id, name, email, emailVerified, image, passwordHash, role, username,
  slug, publicIdentifier, shareCode, timestamps.
- Relationships:
  - accounts
  - sessions
  - profile
  - shareLinks
  - analytics
- Constraints:
  - email unique
  - username unique
  - slug unique
  - publicIdentifier unique
  - shareCode unique
- Indexes:
  - role
  - slug
  - username
  - publicIdentifier

Account
- Purpose: Auth.js OAuth/account compatibility.
- Current credentials flow does not depend on external OAuth, but the model is
  present for Auth.js adapter readiness.
- Composite primary key: provider + providerAccountId.

Session
- Purpose: Auth.js database session compatibility.
- Current Auth.js configuration uses JWT strategy, but the model remains for
  adapter completeness and future session strategy changes.

VerificationToken
- Purpose: Auth.js verification/email flow readiness.

PortfolioProfile
- Purpose: Root CMS entity for one user's portfolio.
- Owns hero, about, projects, skills, experiences, contact, social links, and
  media assets.
- userId is unique to enforce one profile per user.
- isPublic controls public visibility.

HeroSection
- Purpose: Hero content for first viewport.
- One-to-one with PortfolioProfile.
- Contains title, subtitle, CTAs, image URL, badge label/title, order, and
  visibility flag.

AboutSection
- Purpose: About section content.
- One-to-one with PortfolioProfile.
- details is JSON to support label/value rows.
- abilities and progress are string arrays.

Project
- Purpose: Portfolio project cards.
- Many-to-one with PortfolioProfile.
- tags are string array.
- Includes featured, visibility, and order fields.
- Indexes optimize visible ordered profile queries.

Skill
- Purpose: Future structured skill model.
- Currently about abilities are rendered from AboutSection, but Skill exists for
  future richer skill dashboards.

Experience
- Purpose: Shared model for services, education, and general experience.
- type determines usage:
  - service
  - education
  - experience
- Includes icon, organization, period, actionLabel, startsAt, endsAt, order,
  visibility, and featured flags.

ContactInfo
- Purpose: Contact CTA and footer information.
- One-to-one with PortfolioProfile.

SocialLink
- Purpose: External social/profile links.
- Many-to-one with PortfolioProfile.
- Ordered and visibility-aware.

MediaAsset
- Purpose: Gallery and future uploaded media.
- Supports IMAGE, VIDEO, DOCUMENT, OTHER.
- Includes provider and publicId for future Cloudinary integration.

ShareLink
- Purpose: Human-readable public portfolio share URLs.
- Belongs to User.
- Supports customAlias, active toggle, expiration, visit count, and last-opened
  timestamp.
- Indexes userId + active and expiresAt.

VisitorAnalytics
- Purpose: Basic share-link analytics.
- Stores path, referrer, user agent, hashed IP, and timestamp.
- Belongs optionally to User and ShareLink.

7.3 Ownership Rules
-------------------
- User owns PortfolioProfile.
- PortfolioProfile owns content.
- User owns ShareLink.
- VisitorAnalytics may reference User and ShareLink, but uses SetNull on delete
  to avoid destructive analytics coupling.

8. AUTHENTICATION SYSTEM
======================================================================

8.1 Login and Signup
--------------------
Signup page:
- /auth/signup
- Uses AuthForm in signup mode.
- Calls signupAction.
- Validates email/password/name with zod.
- Hashes password with bcryptjs using cost 12.
- Creates User and initial PortfolioProfile with default content.
- Signs in with credentials and redirects to /dashboard.

Login page:
- /auth/login
- Uses AuthForm in login mode.
- Calls loginAction.
- Auth.js Credentials provider checks email and bcrypt password.
- Redirects to /dashboard on success.

Logout:
- logoutAction calls signOut and redirects to /.

8.2 Session Handling
--------------------
Auth.js configuration:
- JWT session strategy.
- maxAge: 30 days.
- Secure production cookie name uses __Secure- prefix.
- Cookie is httpOnly, sameSite lax, path /, and secure in production.

Session callback adds:
- user.id
- user.role
- user.username

JWT callback stores:
- role
- username

8.3 Dashboard Protection
------------------------
Dashboard protection occurs in two places:
- middleware.js checks session-cookie presence before allowing /dashboard.
- dashboard/page.js calls auth() and redirects to /auth/login if no session.

The route-level auth() check is authoritative. Middleware is an early redirect
optimization.

8.4 Role Handling
-----------------
UserRole enum supports USER and ADMIN. Current dashboard access only requires an
authenticated session. Role-based admin policy is prepared but not yet enforced
in dashboard routes.

10. SHAREABLE PORTFOLIO LINK SYSTEM
======================================================================

10.1 URL Strategy
-----------------
The system supports premium human-readable portfolio URLs:
- /portfolio/[slug]
- /u/[username]
- /view/[identifier]

Examples:
- /portfolio/john-doe
- /u/alex-tamang
- /view/john-doe-a12f
- /view/custom-client-link

10.2 Slug Fields
----------------
User model includes:
- username
- slug
- publicIdentifier
- shareCode

ShareLink includes:
- slug
- customAlias
- title
- note
- active
- expiresAt
- visitCount
- lastOpenedAt

10.3 Generation Logic
---------------------
createShareLinkAction:
- Reads customAlias from form.
- Slugifies input.
- Falls back to provided slug, user name, user email, or user id suffix.
- Appends shortSuffix(user.id) to reduce collision risk.
- Stores customAlias when provided.

10.4 Validation Flow
--------------------
/view/[identifier]
  -> resolveShareLink(identifier)
  -> find ShareLink by slug or customAlias
  -> reject if missing
  -> reject if active is false
  -> reject if expiresAt is in the past
  -> increment visitCount
  -> update lastOpenedAt
  -> create VisitorAnalytics
  -> load owner portfolio
  -> render ComponentsPage

Invalid or expired links redirect to:
- /access-denied

10.5 Analytics
--------------
VisitorAnalytics stores:
- userId
- shareLinkId
- path
- referrer
- userAgent
- ipHash
- createdAt

The IP hash uses AUTH_SECRET or NEXTAUTH_SECRET as part of the hash input.

11. ANIMATION SYSTEM
======================================================================

11.1 Animation Principles
-------------------------
The public experience should feel like one uninterrupted cinematic sequence:
- no hard section boundaries
- overlapping visual layers
- persistent ambient background
- section crossfades
- global parallax
- pinned hero
- progressive reveals before full viewport entry

11.2 GSAP Registration
----------------------
src/app/lib/gsapClient.js:
- imports gsap and ScrollTrigger.
- registers ScrollTrigger once.
- configures ignoreMobileResize.
- provides prefersReducedMotion helper.
- provides sectionTrigger helper.

11.3 Hero Pinning
-----------------
usePinnedSection:
- Creates pinned ScrollTrigger timeline.
- Pins hero from top.
- Scrubs media scale/blur.
- Fades and moves content upward.
- Reduces overlay opacity.
- Uses cleanup via ctx.revert().

11.4 Scroll Reveals
-------------------
useScrollReveal:
- js-reveal: fades, translates, and unblurs text/content.
- js-image-reveal: fades, scales, unblurs, and restores saturation.
- js-mask-reveal: animates CSS mask and clip-path circle.
- Reveals start early, before section is fully in viewport.

11.5 Continuous Parallax
------------------------
useParallax:
- Selects [data-speed] layers.
- Moves layers against full page scrollTrigger from top to bottom.
- Desktop and mobile use different travel distances.
- Avoids section-local parallax resets.

11.6 Cinematic Scrub and Section Linking
----------------------------------------
useCinematicScrub:
- Animates global ambient CSS variable --scene-progress.
- Moves/fades light leaks.
- Scrubs js-scrub elements.
- Crossfades outgoing section while incoming section begins early.
- Applies clip-path diagonal wipe transitions.

11.7 Smooth Scrolling
---------------------
useSmoothScroll:
- Custom inertia implementation.
- Uses requestAnimationFrame.
- Intercepts wheel and touchmove.
- Uses lerp toward target scroll position.
- Calls ScrollTrigger.update().
- Handles anchor clicks.
- Cleans listeners on unmount.

11.8 Ambient Layers
-------------------
AmbientMotion and globals.css define:
- fixed cinematic ambient root
- animated color fields
- glows
- fog strips
- particles
- grain
- vignette
- light leaks

11.9 Performance Practices
--------------------------
- Transform/opacity/filter based animations.
- will-change hints for animated classes.
- GSAP contexts for cleanup.
- matchMedia for mobile-specific parallax.
- reduced-motion escape paths.

13. ENVIRONMENT VARIABLES
======================================================================

Required variables:

DATABASE_URL
- Runtime PostgreSQL connection string.
- For Neon, this may use the pooled -pooler host.

DIRECT_URL
- Direct PostgreSQL connection string used by Prisma migrations.
- For Neon, this should not use the -pooler host.

NEXTAUTH_SECRET
- Optional legacy compatibility secret. The application uses AUTH_SECRET as the
  primary source of truth internally. If this is set, it should match
  AUTH_SECRET, but new deployments should prefer AUTH_SECRET.

AUTH_SECRET
- Primary Auth.js secret. This is the single shared secret used internally for
  Auth.js session/JWT signing and analytics IP hashing. It must be stable across
  restarts and deployments. Generate with:
  pnpm secret:generate

AUTH_TRUST_HOST
- Allows Auth.js to trust the deployment host/proxy environment. Required for
  hosted environments such as Vercel when Auth.js cannot infer the trusted host
  safely.

CLOUDINARY_URL
- Reserved for future media upload/storage integration.

NEXT_PUBLIC_APP_URL
- Public application URL used for share-link display and deployment-aware
  generated links.

14.2 Admin Portfolio API
------------------------
Route:
  /api/admin/portfolio

GET:
- Requires authenticated session.
- Rate limit: 60 requests per minute per user/method.
- Returns normalized portfolio data for current user.

PATCH:
- Requires authenticated session.
- Validates payload:
  - displayName: optional string 2-80
  - headline: optional nullable string max 160
  - bio: optional nullable string max 2000
  - location: optional nullable string max 120
  - avatarUrl: optional nullable string max 500
- Updates PortfolioProfile for current user.

14.3 Admin Projects API
-----------------------
Route:
  /api/admin/projects

POST:
- Requires authenticated session.
- Rate limit: 40 requests per minute per user/method.
- Validates payload:
  - title: string 2-120
  - description: string 3-1500
  - imageUrl: optional nullable string max 500
  - tags: string array, each max 40
  - featured: boolean
- Creates Project under authenticated user's profile.

14.4 Server Actions
-------------------
Server actions are used for form-based dashboard and auth flows rather than
pure JSON APIs. They perform auth checks, validation, Prisma writes, and route
revalidation.


======================================================================
15. SECURITY ARCHITECTURE
======================================================================

15.1 Password Security
----------------------
- Passwords are never stored directly.
- signupAction hashes passwords with bcryptjs cost factor 12.
- login compares submitted password with stored passwordHash.

15.2 Session Security
---------------------
- Auth.js JWT session strategy.
- 30-day max age.
- Secure cookie enabled in production.
- httpOnly cookie prevents client-side JavaScript access.
- sameSite lax reduces CSRF exposure for typical navigation.

15.3 Route Protection
---------------------
- /dashboard protected by middleware cookie check.
- dashboard/page.js performs auth() session validation.
- Admin APIs perform auth() validation.
- Server actions perform auth() validation through requireUser().

15.4 CSRF Considerations
------------------------
- Auth.js handles auth route CSRF behavior.
- Server actions are framework-mediated and tied to same-origin application
  flows.
- Future public JSON mutation endpoints should add stricter CSRF/rate-limiting
  controls if exposed.

15.5 Validation
---------------
- zod validates credentials and several dashboard/API payloads.
- JSON section editing currently has basic structure checks and should be
  hardened with section-specific schemas.

15.6 Rate Limiting
------------------
- src/lib/security.js implements in-memory rate limiting.
- Current API routes use per-user/method keys.
- Production multi-instance deployments should move rate limiting to a durable
  shared store such as Redis, Upstash, or Vercel KV.

15.7 Data Ownership
-------------------
- Dashboard actions derive profileId from authenticated user id.
- Share-link toggles use updateMany with id and userId to prevent cross-user
  writes.
- Admin APIs use authenticated user id to scope queries.

--------------------

The project includes vercel.json with these values.

18.3 Prisma Commands
--------------------
Local development migration:
  pnpm prisma migrate dev --name init

Production migration:
  pnpm prisma migrate deploy

Generate client:
  pnpm prisma generate

18.4 Neon Notes
---------------
Use DATABASE_URL for runtime. It may use the pooled host:
  ...-pooler...

Use DIRECT_URL for migrations. It should use the non-pooler host:
  no "-pooler" in hostname

Do not run prisma db pull against a fresh empty database. It will return P4001
because there are no tables to introspect.
